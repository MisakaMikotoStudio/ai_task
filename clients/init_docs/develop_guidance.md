# 开发规范

> 本文档是项目统一开发规范，所有代码开发必须遵守。AI agent 在执行开发任务前必须先读取本文档。

---

## 1. 数据库设计

### 1.1 必备字段

所有数据库表必须包含以下三个字段：

| 字段 | 说明 |
|------|------|
| `created_at` | 记录创建时间，创建时自动填充，不可修改 |
| `updated_at` | 记录更新时间，每次更新自动刷新 |
| `deleted_at` | 记录删除时间，NULL 表示未删除 |

**SQL 示例（MySQL）**：
```sql
`created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
`updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
`deleted_at` DATETIME NULL DEFAULT NULL
```

**SQL 示例（PostgreSQL）**：
```sql
created_at TIMESTAMP NOT NULL DEFAULT NOW(),
updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
deleted_at TIMESTAMP NULL DEFAULT NULL
```
> PostgreSQL 没有 `ON UPDATE` 语法，需通过触发器或 ORM 层维护 `updated_at`。

### 1.2 软删除

- **默认使用软删除**：通过设置 `deleted_at` 字段实现删除，而非物理删除记录
- 所有查询语句默认加上 `WHERE deleted_at IS NULL` 条件
- 仅在以下情况允许硬删除：明确的设计要求（如降低存储压力、合规要求清除数据）
- 硬删除必须在开发文档中说明原因

### 1.3 数据归属

- 每张表必须有明确的**归属字段**标识记录所有者
- 对于需要用户登录的系统，每张表必须包含 `user_id` 字段
- **所有 SQL 操作（查询、更新、删除）必须携带归属字段条件**，防止越权访问
- 禁止出现不带归属字段过滤的数据操作，避免用户 A 看到或修改用户 B 的数据

### 1.4 时区处理

- 数据库时间字段统一使用 **UTC+0** 时间存储
- 代码中所有时间操作统一使用 UTC 时间，不依赖服务器本地时区
- 时区转换仅在展示层（前端）进行
- 考虑以下跨时区场景：数据库多实例部署、后端服务器分布在不同时区、前端用户处于不同时区

### 1.5 性能优化

- **禁止在循环中执行单条 SQL**：需要操作多条记录时，必须使用批量操作（batch insert/update）
- **批量操作需控制批次大小**：单次批量操作建议不超过 500-1000 条，避免单条 SQL 执行过慢或锁表时间过长
- **合理使用索引**：WHERE 条件中的高频查询字段必须建立索引，联合查询考虑组合索引
- **避免 SELECT ***：仅查询所需字段，减少数据传输和内存消耗
- **大数据量查询使用分页**：禁止一次性加载全表数据，使用 LIMIT/OFFSET 或游标分页

## 2. 代码开发

### 2.1 日志规范

**必须打印日志的场景**：
- 外部服务调用（请求和响应）
- 关键业务操作（创建、修改、删除）
- 异常和错误处理
- 定时任务执行

**日志必须包含的信息**：
| 信息 | 说明 |
|------|------|
| 时间戳 | UTC+0 时间，与数据库时间保持一致 |
| 文件名及行号 | 便于快速定位问题 |
| 日志级别 | DEBUG / INFO / WARN / ERROR |
| 请求标识 | request_id 或 trace_id，用于串联一次请求的完整链路 |
| 异常栈 | ERROR 级别日志必须包含完整异常堆栈 |

**日志禁忌**：
- 禁止在日志中打印密码、token、密钥等机密信息
- 禁止在循环中打印大量重复日志
- 禁止仅打印 "error occurred" 等无上下文信息的日志

### 2.2 机密信息管理

- **禁止在代码中硬编码**任何机密信息（密码、API Key、密钥、token 等）
- 机密信息必须通过以下方式管理：服务启动配置文件、环境变量、密钥管理服务
- 代码仓库中不得提交包含真实机密信息的配置文件，应使用 `.example` 模板文件
- `.gitignore` 中必须排除包含机密信息的配置文件

### 2.3 函数调用规范

- **所有函数调用必须使用命名参数**：`func(param_name=value)`，不要省略参数名
- 目的：避免函数签名变更（参数顺序调整、新增参数）导致调用方出现隐蔽错误

### 2.4 函数设计

- 单个函数不超过 200 行，超出则拆分为子函数
- 函数职责单一，一个函数只做一件事
- 返回值类型保持一致，避免同一函数在不同分支返回不同类型

### 2.5 错误处理

- 不要吞掉异常：`except: pass` 是严格禁止的
- 异常捕获要尽量具体，避免 `except Exception` 大范围捕获
- 错误信息要包含足够的上下文（操作内容、输入参数、失败原因）
- 对于可恢复错误和不可恢复错误，要有明确的处理策略区分

### 2.6 命名规范

- 变量名、函数名必须具有明确的业务含义，禁止使用 `a`、`b`、`tmp`、`data` 等无意义名称
- 布尔变量使用 `is_`、`has_`、`can_` 等前缀
- 常量使用全大写加下划线：`MAX_RETRY_COUNT`
- 命名风格在项目内保持一致（如统一使用 snake_case 或 camelCase）

## 3. HTTP 接口开发

### 3.1 统一拦截器（中间件）

所有 HTTP 接口必须通过统一拦截器，拦截器负责以下职责：

**请求/响应日志**：
- 记录请求方法、路径、耗时、响应状态码
- 请求体和响应体可选记录（注意大小限制）
- **禁止记录鉴权头（Authorization）、Cookie 中的 token 等机密信息**

**登录鉴权**：
- 默认所有接口都需要鉴权
- 不需要鉴权的接口必须显式标注（如 `@skip_auth` 注解/装饰器）
- 鉴权失败统一返回 401 状态码

**异常统一处理**：
- 所有异常在拦截器层统一捕获和封装，**确保返回给调用方的永远是标准 JSON 格式**
- 具体接口实现内部不需要重复做 try-catch 处理
- 标准错误响应格式：

```json
{
  "code": 40001,
  "message": "人类可读的错误描述",
  "detail": "调试用的详细信息（仅非生产环境返回）"
}
```

### 3.2 接口设计规范

- 使用 RESTful 风格：GET 查询、POST 创建、PUT 更新、DELETE 删除
- 路径使用小写加连字符：`/api/user-profiles`，不使用驼峰或下划线
- 列表接口必须支持分页，返回总数和分页信息
- 接口版本管理：通过 URL 前缀区分版本（如 `/api/v1/`）

### 3.3 请求参数校验

- 所有输入参数必须进行校验，不信任任何客户端输入
- 使用框架提供的参数校验机制（如 Pydantic、marshmallow），避免手动校验
- 校验失败返回 400 状态码，并明确说明哪个参数不合法

## 4. 安全规范

### 4.1 输入安全

- 所有外部输入（HTTP 参数、文件上传、第三方回调）必须做合法性校验
- SQL 操作使用参数化查询，禁止拼接 SQL 字符串
- 文件上传需校验文件类型和大小

### 4.2 权限控制

- 数据操作必须校验当前用户是否有权限操作目标数据（参见 1.3 数据归属）
- 敏感操作（删除、修改权限）需要额外的权限校验
- API 接口避免暴露内部 ID 序列，考虑使用 UUID 或加密 ID

### 4.3 依赖安全

- 禁止引入来源不明的第三方依赖
- 定期检查依赖项是否存在已知安全漏洞

---

## 5. 配置管理

### 5.1 配置分层

- **代码中的默认值**：合理的默认行为，确保缺少配置时不会崩溃
- **配置文件**：环境相关的配置（数据库地址、服务端口等）
- **环境变量**：敏感配置和部署环境差异化配置

### 5.2 配置规范

- 配置项命名清晰，使用统一的命名前缀区分模块
- 必须提供配置文件模板（如 `config.example.yaml`），注释说明每个配置项的用途和取值范围
- 关键配置项缺失时程序应在启动阶段报错退出，不要等到运行时才失败

---

## 6. 并发与异步

- 共享资源的并发访问必须使用锁或原子操作保护
- 数据库操作中涉及"先查后改"的逻辑，要考虑并发场景下的竞态条件，使用乐观锁或悲观锁
- 异步任务需要有超时机制和重试策略
- 长时间运行的任务要支持优雅中断

---

## 7. 测试规范

- 核心业务逻辑必须有单元测试覆盖
- 单元测试不依赖外部服务（数据库、网络），使用 mock 替代
- 测试用例必须覆盖正常路径和异常路径
- 修改代码后必须运行已有测试，确保不引入回归
