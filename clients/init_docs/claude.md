# CLAUDE.md

## 1. 工作环境说明

**⚠️ 多仓库环境**: 当前工作目录（$CWD）**不是**一个git仓库，$CWD 下的多个子文件夹是独立的git仓库项目，开发任务可能跨越多个仓库。

**关键规则**:
- 操作文件前必须明确该文件属于哪个仓库
- 跨仓库任务需要分别了解各仓库的上下文

### 变量定义

```
$CWD      = 当前工作目录（非git仓库，包含多个git仓库子目录）
$REPO     = $CWD 下某个git仓库子目录（如 $CWD/repo_name/）
$GUIDANCE = $CWD/develop_guidance.md  （开发规范，工作目录级别）
$DEVELOP  = $CWD/develop.md           （需求开发文档，工作目录级别）
```

### 确认当前环境

```bash
# 列出工作目录下所有git仓库
find . -maxdepth 2 -name ".git" -type d | xargs -I {} dirname {}
```

## 2. 工作流程

**执行模式**: SEQUENTIAL（按顺序执行）

### STEP 1 [REQUIRED]: 读取开发规范

**目的**: 确保开发工作符合项目统一规范，避免引入不一致的代码风格和设计

**动作**:
```bash
cat $CWD/develop_guidance.md
```

**要求**:
- 在写任何代码前，必须先读取并理解开发规范
- 开发过程中严格遵守规范中的各项约定
- 如果当前任务与规范存在冲突，在开发文档中说明原因

---

### STEP 2 [REQUIRED]: 创建需求开发文档

**动作**: 创建 `develop.md`

**模板参考**: [develop_plan_example.md]($CWD/develop_plan_example.md)

**目的**:
- 给用户提供整体任务开发概览
- 用户通过此文档了解AI做了什么、为什么这样做
- AI与用户的交互载体

**信息不足时**: 在文档开头添加TODO列表，告知用户需要补充的信息。**同时，agent 执行结束后，必须将待确认事项作为返回内容的一部分展示给调用方**，而不仅仅写在开发文档中，确保调用方能第一时间看到需要补充的信息。

**跨仓库任务**: 在文档中明确列出各仓库的改动范围和依赖关系

---

### STEP 3 [REQUIRED]: 执行开发

**要求**:
- 严格按照开发文档中的方案执行
- 每完成一个模块，更新开发文档中的进度状态
- 遇到方案需要调整时，先更新开发文档再改代码

---

## 3. 代码规范

> 以下为基础规范，完整规范参见 `$GUIDANCE`（develop_guidance.md）。

1. **遵守开发规范**: 所有代码必须符合 `develop_guidance.md` 中定义的规范

2. **单测要求**: 如项目已有单测框架，按现有规范添加单元测试；即使不需要新增单测，也需执行已有单测确保不引入回归

3. **兼容性设计**:
   - 新方案上线后，老功能必须正常工作
   - 考虑回滚场景，确保回滚时不报错
   - 无法避免时，在开发文档风险章节中说明

4. **函数长度**: 单个函数不超过200行，超出则拆分

## 4. Git规范

### 仓库同步

当要求"同步云端"或"强制同步云端最新内容"时：
```bash
git fetch origin && git reset --hard origin/<分支名>
```
⚠️ 这会丢弃本地所有未提交的更改

### 安全协议

| 操作 | 规则 |
|-----|-----|
| 更新git配置 | ❌ 永远不要 |
| 强制推送 | ❌ 除非明确要求 |
| 强制推送到main/master | ❌ 绝对禁止 |
| --amend | ❌ 除非用户要求 |
| 硬重置 | ❌ 除非明确要求 |

- [ ] 我理解"为什么"再去"怎么做"了吗？

**跨仓库检查**（如涉及多仓库）:
- [ ] 我确认了当前工作目录下有哪些仓库吗？
- [ ] 我在开发文档中记录了各仓库的改动范围吗？
- [ ] 我分析了跨仓库的依赖关系吗？

**如果任何一个答案是"否"，停下来先解决它。**
